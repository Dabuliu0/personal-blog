# 文章修改设计

## 修改流程

用户点击编辑：

↓

获取文章 ID

↓

查询文章

↓

修改内容

↓

提交保存

↓

后端验证

↓

修改文章

## 前端和后端职责

前端负责表达用户意图。

后端负责： - 数据校验 - 权限判断 - 业务规则

## 更新方式

### 方案一：部分更新

只提交修改字段。

例如：

``` json
{
    "title":"Java进阶"
}
```

优点： 灵活。

缺点： 需要处理 null 和空字符串。

### 方案二：完整更新

提交完整文章数据。

优点： 简单。

缺点： 字段多时麻烦。

### 方案三：明确修改状态

额外记录： - 是否修改 - 修改后的值

可以区分： - 没修改 - 修改为空 - 修改为某个值

## null 和空字符串

null：

表示没有提交这个字段。

空字符串：

表示提交了修改，但是内容为空。

## 我的理解

修改功能不是简单 CRUD。

真正需要考虑：

-   谁可以修改？
-   修改什么？
-   什么时候允许修改？
-   修改后是否合法？

后端开发就是围绕这些规则组织代码。
